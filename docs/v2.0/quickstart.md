# Quickstart: Choose Your Deployment Path

LibreQoS supports multiple real-world deployment paths. Start by selecting one path and keep a single source-of-truth model for shaping data.

## Before You Start

- WebUI is the primary operational interface after install.
- Pick one source-of-truth model for `network.json` and `ShapedDevices.csv`.
- Avoid competing edits across multiple systems for the same objects.

## Testbed / Lab

Use this path when you want to validate LibreQoS behavior in a controlled environment before inline production deployment.

### Lab setup flow

1. Build a lab topology (LibreQoS box + traffic generator endpoints + optional routers for OSPF/BGP path simulation).
2. Complete platform and install prerequisites:
   - [Deployment Scenarios](design.md)
   - [System Requirements](requirements.md)
   - [Server Setup - Prerequisites](prereq.md)
   - [Install Ubuntu Server 24.04](ubuntu-server.md)
   - [Configure Shaping Bridge](bridge.md)
3. Install LibreQoS using `.deb` package (recommended):

```bash
cd ~
sudo apt-get update
sudo apt-get upgrade
wget https://download.libreqos.com/{deb_url_v1_5}
sudo apt install ./{deb_url_v1_5}
```

4. Choose one test track:
   - **Lab with manual files**: use sample/manual `network.json` and `ShapedDevices.csv`.
   - **Lab with supported integration**: configure integration and validate pulled data.
5. Generate test traffic and validate in WebUI:
   - Dashboard/Tree/Flow behavior
   - Scheduler status
   - Urgent issues
6. When behavior is as expected, move to an inline pilot path.

## Supported Integration

Use this path when your CRM/NMS is supported by built-in LibreQoS integrations.

1. Complete install prerequisites and install LibreQoS (same links/steps as above).
2. Configure integration in WebUI.
3. Validate imported shaping data and topology in WebUI.
4. Place the LibreQoS box inline for pilot traffic.
5. Monitor scheduler health, shaping behavior, and urgent issues in WebUI.
6. Expand from pilot to broader deployment after stable observations.

Note:
- In integration mode, `ShapedDevices.csv` is typically regenerated by sync jobs.
- `network.json` overwrite behavior depends on integration settings (for example `always_overwrite_network_json`).

## Custom Script Integration

Use this path when your CRM/NMS is unsupported and you will generate `network.json` and `ShapedDevices.csv` with your own scripts.

1. Complete install prerequisites and install LibreQoS (same links/steps as above).
2. Implement script/process to generate and refresh `network.json` and `ShapedDevices.csv`.
3. Treat script outputs as source-of-truth for long-term persistence.
4. Place the LibreQoS box inline for pilot traffic.
5. Use WebUI for operational checks and quick edits.
6. Move permanent logic/data changes back into your script pipeline.

Note:
- WebUI edits are useful for quick operational changes.
- Long-term state should be maintained by your external source-of-truth workflow.

## Manual Files (<100 subscribers)

Use this path when you intentionally maintain `network.json` and `ShapedDevices.csv` directly without CRM/NMS synchronization.

Recommended only for networks under 100 subscribers.

1. Complete install prerequisites and install LibreQoS (same links/steps as above).
2. Build and maintain `network.json` and `ShapedDevices.csv` directly.
3. Place the LibreQoS box inline for pilot traffic.
4. Use WebUI to validate shaping, scheduler status, and topology behavior.
5. Keep manual update discipline tight (data quality, consistency, update cadence).
6. If subscriber count or change volume grows, plan migration to supported integration or custom script integration.
